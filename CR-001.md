## What Claude Code Security is doing (the parts worth copying)

From Anthropic’s own descriptions, the “shape” of Claude Code Security is:

- **Whole‑repo scanning** that “reasons through code like a security researcher,” tracing context and data flow across files rather than only matching patterns.  [oai_citation:0‡Claude](https://claude.com/solutions/claude-code-security)  
- **Parallel scanning** (divide the repo into chunks/subsystems and scan concurrently).  [oai_citation:1‡Claude](https://claude.com/solutions/claude-code-security)  
- **Multi‑stage / adversarial verification**: every candidate finding gets a second pass that tries to *disprove* the finding to reduce false positives.  [oai_citation:2‡Claude](https://claude.com/solutions/claude-code-security)  
- **Severity + confidence ratings**, so teams can triage.  [oai_citation:3‡Anthropic](https://www.anthropic.com/news/claude-code-security)  
- **Patch proposals for human review**, with an explicit “nothing is applied without human approval” stance.  [oai_citation:4‡Claude](https://claude.com/solutions/claude-code-security)  

That’s the blueprint to implement in Codex CLI: **scan → verify → propose patch → export**, with “apply” as an explicit opt‑in.

# Change request: Add `codex security scan` (whole-repo scan → verify → patch proposals)

## 1) CLI UX

Add a stable command group:

- `codex security scan`  
  - default: **scan-only**, prints Markdown summary + detailed findings
  - optional: `--format json|sarif|all`, `--out …`
  - optional: `--apply` gated behind explicit confirmations (never default)

Recommended defaults:
- sandbox: **read-only**
- approvals: **never**
- network: **off**
- web search: **disabled** (or cached-only)  [oai_citation:25‡OpenAI Developers](https://developers.openai.com/codex/security/)  

This makes scanning safe on “random GitHub clones” by default.

## 2) Core workflow (Claude-like pipeline)

Implement the pipeline explicitly in code:

1) **Repo fingerprint + map**
   - languages + framework detection (package manifests)
   - entrypoints (routes/controllers/handlers)
   - “sinks” inventory (SQL exec, command exec, template render, file writes, deserialization)
2) **Parallel candidate discovery**
   - split by subsystem (auth, routing, DB, templates) and/or vulnerability class
3) **Adversarial verification pass**
   - second pass tries to disprove each candidate
4) **Patch proposal**
   - minimal diffs, style-preserving
   - never auto-apply unless `--apply` + confirmation
5) **Export**
   - Markdown + JSON
   - SARIF output for CI/code scanning

This mirrors Claude’s “scan in parallel” + “adversarial verification pass” + “patches for human approval” approach.  [oai_citation:26‡Claude](https://claude.com/solutions/claude-code-security)  

## 3) Output contract (make it robust)

Use a strict JSON schema for tool output:
- findings array
- `status`: candidate|confirmed|needs-manual-review|dismissed
- severity + confidence
- file/line evidence
- patch diff (optional)

Codex already supports validating the final response against a JSON Schema file via `--output-schema`.  [oai_citation:27‡OpenAI Developers](https://developers.openai.com/codex/cli/reference/)  
So your subcommand can:
- ship `security_scan.schema.json` inside the repo/binary
- run the scan agent with `--output-schema <thatfile>`
- parse the validated JSON to render Markdown and/or SARIF

## 4) Skills & “install everything” story

### Make it “installed by default”
Bundle the `security-scan` skill as a **SYSTEM** skill (same category as Codex’s built-ins). Codex’s skills docs explicitly call out “SYSTEM: bundled with Codex” and “available to everyone when they start Codex.”  [oai_citation:28‡OpenAI Developers](https://developers.openai.com/codex/skills/)  

That meets your “no setup” requirement: the scan feature is simply there.

### Still support a one-liner install
If you also want optional add-ons (extra playbooks, organization rules, etc.), keep compatibility with `$skill-installer`.  [oai_citation:29‡OpenAI Developers](https://developers.openai.com/codex/skills/)  

In practice:
- ship baseline `security-scan` as system skill
- optionally ship “language playbooks” as part of the same system skill (recommended)
- later: allow `$skill-installer install <your-security-pack>` for extra rulesets

## 5) Language/framework packs (how to cover “popular stacks” without endless bespoke code)

Use a “pack” model:

- One top-level skill (`security-scan`) that defines the workflow and output contract.
- Under `references/`, maintain small playbooks:
  - `react.md` (DOM XSS pitfalls, `dangerouslySetInnerHTML`, unsafe URL handling, etc.)
  - `nextjs.md` (API routes, server actions, auth/session patterns)
  - `node.md` (Express/Nest sinks: `child_process`, SSRF, prototype pollution, template injection)
  - `python-web.md` (FastAPI/Flask/Django sinks: SQL injection via string concat, SSRF via requests, pickle, etc.)
  - `.net.md` (ASP.NET Core sinks: deserialization, authz filters, path traversal)
  - `spring.md` (Spring Boot sinks: deserialization gadgets, SpEL injection, authz)
- In Rust, keep detection logic simple:
  - framework detection by manifest dependencies (e.g., `package.json` contains `react`/`next`)
  - entry point heuristics (e.g., `app/api/**` for Next)
- Let the model do the deeper “trace flow across files” reasoning.

This gives you a maintainable path to “most popular stacks” without building a full Semgrep replacement.

## 6) “Modify itself” support (safely)

Two things matter here:

1) **Dogfooding**: Codex should be able to run `codex security scan` against the Codex repo itself and propose fixes (that’s the practical meaning of “modify itself”).  
2) **Guardrails**: applying fixes must remain human-approved, and the command must respect Codex’s sandbox/approval model.  [oai_citation:30‡OpenAI Developers](https://developers.openai.com/codex/security/)  

What I’d implement:
- `codex security scan` is always safe/read-only by default.
- `codex security fix --id CSS-001` applies a *single* fix at a time:
  - creates a branch/worktree
  - applies diff
  - runs tests (optional)
  - asks for confirmation at each destructive step

Also note: Codex has specific cyber-safety safeguards for high-capability cybersecurity behavior; some requests may be routed to a less cyber-capable model unless the account has “Trusted Access for Cyber.” If you build security scanning into your fork and run it heavily, this is something you may bump into.  [oai_citation:31‡OpenAI Developers](https://developers.openai.com/codex/concepts/cyber-safety/)
